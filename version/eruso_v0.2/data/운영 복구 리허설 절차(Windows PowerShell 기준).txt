아래는 요청하신 운영 복구 리허설 절차를 Windows PowerShell 기준으로 “단계 ↔ 명령어” 1:1 매칭한 것입니다. (현재 wrangler 4.65.0에서는 d1 export / d1 execute / d1 create / d1 list / d1 info가 유효합니다. 또한 Pages 바인딩은 DB → eruso-dds-db로 연결된 화면을 이미 확인했습니다.) Source


로그확인방법 : C:\Users\user\AppData\Roaming\xdg.config\.wrangler\logs

0) 사전 준비(권장)
작업 폴더 하나 만들기(백업 파일 저장 위치)
Copymkdir C:\d1-backups -Force
cd C:\d1-backups
(선택) API 토큰 환경변수 세팅(이미 세팅돼 있으면 생략)
Copy$env:CLOUDFLARE_API_TOKEN="YOUR_TOKEN"



1) “지금 운영 DB(eruso-dds-db)를 export로 백업 생성”
✅ 명령어
Copy$ts = Get-Date -Format "yyyyMMdd-HHmmss"
npx wrangler d1 export eruso-dds-db --remote --output ".\eruso-dds-db-$ts.sql"
백업 파일 생성 확인:
Copydir .\eruso-dds-db-*.sql | select -last 1
(Export는 D1 DB 전체를 .sql로 내보내는 공식 방식입니다.) Source

2) “새 D1 DB를 하나 만들고(예: eruso-dds-restore-test)”
✅ 명령어
Copynpx wrangler d1 create eruso-dds-restore-test
생성 후 목록에서 확인:
Copynpx wrangler d1 list
참고: create 결과로 새 DB의 database_id가 출력됩니다. 나중에 Pages 바인딩 전환할 때 유용합니다.

3) “그 새 DB에 백업 SQL을 import(d1 execute --file)”
✅ 명령어
방금 만든 백업 파일명을 변수로 잡고(가장 최근 파일)
Copy$backup = (Get-ChildItem .\eruso-dds-db-*.sql | Sort-Object LastWriteTime -Descending | Select-Object -First 1).FullName
$backup
새 DB로 import 실행
Copynpx wrangler d1 execute eruso-dds-restore-test --remote --file "$backup"
(Import는 d1 execute --file로 SQL을 실행하는 방식이 공식 가이드에 있습니다.) Source

4) “몇 개 핵심 쿼리로 검증(users/deliveries/prescriptions count)”
✅ 명령어(새 DB에서 검증)
(A) 테이블별 카운트 한 번에
Copynpx wrangler d1 execute eruso-dds-restore-test --remote --command `
"SELECT 'users' AS table_name, COUNT(*) AS cnt FROM users
 UNION ALL SELECT 'deliveries', COUNT(*) FROM deliveries
 UNION ALL SELECT 'prescriptions', COUNT(*) FROM prescriptions;"
(B) 샘플 레코드 5건씩(정상 데이터인지 빠른 육안 확인)
Copynpx wrangler d1 execute eruso-dds-restore-test --remote --command "SELECT * FROM users ORDER BY rowid DESC LIMIT 5;"
npx wrangler d1 execute eruso-dds-restore-test --remote --command "SELECT * FROM deliveries ORDER BY rowid DESC LIMIT 5;"
npx wrangler d1 execute eruso-dds-restore-test --remote --command "SELECT * FROM prescriptions ORDER BY rowid DESC LIMIT 5;"

5) “운영 복구가 필요할 때만 Time Travel 또는 바인딩 전환을 선택”
옵션 A) Time Travel로 운영 DB를 특정 시점으로 복구
(실수로 DELETE/UPDATE, 마이그레이션 실패 등 “운영 DB 자체를 되돌려야 할 때”)
Time Travel은 분 단위 시점 복구를 지원합니다. Source

✅ 명령어
현재 북마크/정보 확인:
Copynpx wrangler d1 time-travel info eruso-dds-db
특정 시점으로 복구(예시 시각은 상황에 맞게):
Copynpx wrangler d1 time-travel restore eruso-dds-db --timestamp "2026-02-16T00:10:00+09:00"
옵션 B) Pages 바인딩을 “복구 검증된 DB(eruso-dds-restore-test)”로 전환
이건 “운영 DB가 망가졌고, 복구 DB를 새 운영 DB로 갈아끼우는” 전략입니다.

✅ 실제 작업 흐름(권장)
eruso-dds-restore-test가 정상인지(4번) 검증 완료
Cloudflare Dashboard → Pages → 프로젝트 eruso-dds → Settings → Bindings
D1 database binding(이름: DB)의 값을 eruso-dds-db → eruso-dds-restore-test로 변경
현재 바인딩이 DB → eruso-dds-db인 화면은 이미 확인됨 Source
저장 후(필요 시) 재배포/캐시 무효화 후 서비스 확인
체크포인트(성공 판정)
새 DB(eruso-dds-restore-test)에서 카운트가 운영 DB와 동일/기대치와 일치
사이트에서 사용자 드롭다운/배송 목록 등이 정상 조회
예: 로그인 드롭다운이 사용자 목록을 정상 표시했던 화면 참고 Source
